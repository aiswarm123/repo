# Telegram Support Bot — Посібник користувача

Покроковий посібник для всіх, хто взаємодіє з ботом: кінцеві користувачі, що відкривають заявки,
агенти підтримки, що відповідають на них, та адміністратори, що керують розгортанням.

## 1. Передумови

Перед початком вам знадобляться:

- **Python 3.12+** (при локальному запуску) **або Docker** (при запуску в контейнері)
- Обліковий запис **Telegram**
- Доступ до **@BotFather** — офіційного бота Telegram для створення та керування ботами
- **Виділена група Telegram**, яка слугуватиме вхідною скринькою підтримки (бот пересилає
  туди кожну заявку користувача)

## 2. Встановлення

### ✅ Варіант A — Локально (Python)

1. **Клонуйте або завантажте репозиторій**

   ```bash
   git clone https://github.com/aiswarm123/repo.git
   cd repo/examples/telegram-support-bot
   ```

2. **Створіть віртуальне середовище та встановіть залежності**

   ```bash
   python -m venv .venv
   source .venv/bin/activate          # Windows: .venv\Scripts\activate
   pip install -r requirements.txt
   ```

   Або, якщо у вас є `make`:

   ```bash
   make install
   ```

3. **Створіть файл `.env`**

   ```bash
   cp .env.example .env
   ```

   Відкрийте `.env` у будь-якому текстовому редакторі та заповніть значення (деталі у
   [Розділі 3](#3-довідник-конфігурації)).

4. **Запустіть бота**

   ```bash
   python main.py
   ```

   Або за допомогою `make`:

   ```bash
   make run
   ```

   Ви побачите рядки логу `Starting bot...` — бот тепер опитує Telegram.

---

### ✅ Варіант B — Docker

1. **Створіть файл `.env`**

   ```bash
   cp .env.example .env
   ```

   Заповніть значення (дивіться [Розділ 3](#3-довідник-конфігурації)).

2. **Збудуйте та запустіть контейнер**

   ```bash
   make docker-up
   ```

   Це виконує `docker compose up -d --build` у фоновому режимі.

3. **Перевірте логи**

   ```bash
   make docker-logs
   ```

   У виводі ви маєте побачити `Starting bot...`. Натисніть `Ctrl+C`, щоб зупинити перегляд
   логів — контейнер продовжує працювати.

4. **Зупиніть бота за потреби**

   ```bash
   docker compose down
   ```

## 3. Довідник конфігурації

Усі налаштування зчитуються з файлу `.env` (або реальних змінних середовища).

| Змінна | Обов'язкова | За замовчуванням | Опис та як отримати |
|---|---|---|---|
| `BOT_TOKEN` | **Так** | — | API-токен вашого бота. Відкрийте **@BotFather** → `/newbot` → дотримуйтесь підказок → скопіюйте токен, який він надасть (виглядає як `123456:ABC-DEF…`). |
| `SUPPORT_CHAT_ID` | **Так** | — | Ідентифікатор групи Telegram, яка отримує пересилані заявки. Додайте **@getidsbot** до вашої групи підтримки, надішліть будь-яке повідомлення та скопіюйте **від'ємне** число, яке він повідомить (наприклад, `-1001234567890`). |
| `DATABASE_PATH` | Ні | `support.db` | Шлях до SQLite-файлу, де зберігаються заявки. Змініть, якщо хочете розмістити базу даних в іншому місці (наприклад, `/data/support.db`). |

### Покрокове отримання токена бота

1. Відкрийте Telegram і знайдіть **@BotFather**.
2. Надішліть `/newbot`.
3. Введіть відображуване ім'я бота (наприклад, `Acme Support`).
4. Введіть ім'я користувача, що закінчується на `bot` (наприклад, `acme_support_bot`).
5. BotFather надішле токен — скопіюйте весь рядок і вставте його як `BOT_TOKEN`.

### Отримання ідентифікатора групи підтримки

1. Створіть нову групу Telegram (або використайте існуючу).
2. Додайте самого бота як учасника цієї групи.
3. Тимчасово додайте **@getidsbot** до групи.
4. @getidsbot опублікує ідентифікатор групи — це буде **від'ємне** число (наприклад, `-1001234567890`).
5. Скопіюйте це число як `SUPPORT_CHAT_ID`, потім видаліть @getidsbot.

## 4. Посібник для кінцевого користувача

Користувачі взаємодіють з ботом у **приватному чаті** — ніколи в групі підтримки.

### Запуск бота

Знайдіть бота за ім'ям користувача та натисніть **Старт**, або надішліть `/start`.

```
User:  /start
Bot:   Welcome to Support Bot!

       Use /ticket to open a new support ticket.
       Use /mystatus <ticket_id> to check a ticket's status.
```

---

### 📋 Відкриття заявки

Бот проведе вас через два кроки: короткий заголовок, а потім детальний опис.

```
User:  /ticket
Bot:   Please enter a short subject for your ticket:
       [Cancel]

User:  Can't log in

Bot:   Now describe your issue in detail:
       [Cancel]

User:  I enter my password and keep getting "Invalid credentials"
       even though I'm sure it's correct.

Bot:   Ticket #42 submitted! Our team will get back to you shortly.
       [View ticket status]
```

- Натисніть **Cancel** у будь-який момент, щоб скасувати без створення заявки.
- Натисніть **View ticket status** після подання, щоб переглянути поточний статус у спливаючому вікні.

---

### Перевірка статусу заявки

Використовуйте `/mystatus <ticket_id>` з номером із підтвердного повідомлення.

```
User:  /mystatus 42

Bot:   Ticket #42
       Subject: Can't log in
       Status: open
       Created: 2026-03-01 14:05:32

       Replies:
       • Try resetting your password at example.com/reset
```

Якщо заявка ще не має відповідей, розділ **Replies** опускається.

> ⚠️ Ви можете переглядати лише заявки, відкриті вами особисто. Спроба перевірити
> чужу заявку повертає помилку.

---

### Отримання відповіді від команди підтримки

Коли агент відповідає на вашу заявку, бот автоматично надсилає вам приватне повідомлення —
вам нічого не потрібно робити.

```
Bot:   📬 Reply to your ticket #42

       Try resetting your password at example.com/reset
```

---

### Сповіщення про вирішення заявки

Коли агент позначає вашу заявку як вирішену, ви отримуєте:

```
Bot:   ✅ Your ticket #42 has been resolved.
       Thank you for contacting support!
```

## 5. Посібник для команди підтримки

Агенти підтримки працюють у **групі підтримки**, яку ви налаштували за допомогою `SUPPORT_CHAT_ID`.
Бот публікує там кожну нову заявку та надає команди для відповіді.

### ⚠️ Важливо: бот має бути учасником групи

Бот не публікуватиме заявки в групі, якщо його не **додано як учасника**.
Якщо заявки не з'являються, перевірте це насамперед (дивіться [Усунення несправностей](#7-усунення-несправностей)).

---

### Отримання заявки

Коли користувач подає заявку, бот публікує повідомлення в групі підтримки:

```
Bot:   📩 New ticket #42
       From: @alex123
       Subject: Can't log in

       I enter my password and keep getting "Invalid credentials"
       even though I'm sure it's correct.

       Reply with:   /reply 42 <message>
       Resolve with: /resolve 42
```

---

### Відповідь на заявку

```
Agent: /reply 42 Try resetting your password at example.com/reset

Bot:   Reply sent for ticket #42.
```

Користувач отримує відповідь як приватне повідомлення негайно. Можна надсилати кілька
відповідей на одну заявку.

> ⚠️ Відповідь на вже вирішену заявку не дозволена:
> ```
> Agent: /reply 42 One more tip
> Bot:   Ticket #42 is already resolved.
> ```

---

### Вирішення заявки

```
Agent: /resolve 42

Bot:   Ticket #42 marked as resolved.
```

Користувач отримує сповіщення про те, що його заявку вирішено. Після вирішення заявку
не можна повторно відкрити за допомогою команд бота.

---

### Перегляд усіх відкритих заявок

```
Agent: /open_tickets

Bot:   Open tickets (3):
       • #41 — Login page crashes [@bob456] (/reply 41 ...)
       • #42 — Can't log in [@alex123] (/reply 42 ...)
       • #43 — Payment error [987654321] (/reply 43 ...)
```

Користувачі без публічного імені відображаються за їхнім числовим ідентифікатором Telegram.

## 6. Посібник для адміністратора

Поточна версія бота не постачається зі спеціальними адміністраторськими командами. Адміністратори
керують розгортанням на рівні інфраструктури.

### Запуск і зупинка

```bash
# Локально
make run            # запуск
Ctrl+C              # зупинка

# Docker
make docker-up      # запуск (у фоні)
make docker-down    # зупинка
make docker-logs    # перегляд логів
```

### Перевірка бази даних

Бот використовує базу даних SQLite (за замовчуванням: `support.db`). Ви можете запитувати її
безпосередньо за допомогою будь-якого SQLite-клієнта.

```bash
sqlite3 support.db

# Список усіх відкритих заявок
SELECT id, username, subject, created_at FROM tickets WHERE status = 'open';

# Показати всі відповіді для заявки #42
SELECT body, created_at FROM ticket_replies WHERE ticket_id = 42;

# Вручну позначити заявку як вирішену
UPDATE tickets SET status = 'resolved', resolved_at = CURRENT_TIMESTAMP
WHERE id = 42;
```

### Зміна конфігурації

1. Відредагуйте `.env`.
2. Перезапустіть бота:

   ```bash
   # Локально
   Ctrl+C
   make run

   # Docker
   make docker-down
   make docker-up
   ```

## 7. Усунення несправностей

| Проблема | Можлива причина | Виправлення |
|---|---|---|
| Бот не відповідає на жодне повідомлення | `BOT_TOKEN` неправильний або відкликаний | Повторно скопіюйте токен з @BotFather (`/mybots` → виберіть бота → **API Token**). |
| Заявки не з'являються в групі підтримки | Бот не є учасником групи | Відкрийте групу, натисніть на назву групи → **Додати учасників** → знайдіть свого бота та додайте його. |
| Помилка `SUPPORT_CHAT_ID` при запуску | Значення відсутнє або не є дійсним цілим числом | Ідентифікатор повинен бути **від'ємним** числом (наприклад, `-1001234567890`). Повторно запустіть @getidsbot у групі. |
| "Ticket #X not found" при використанні `/mystatus` | Неправильний ідентифікатор заявки або файл бази даних знаходиться в іншому місці | Перевірте `DATABASE_PATH` у `.env`. Переконайтеся, що ви надсилаєте правильний номер із підтвердного повідомлення. |
| "You don't have permission to view that ticket" | Спроба переглянути заявку іншого користувача | Кожен користувач може переглядати лише свої заявки. |
| Бот запускається, але одразу падає | Відсутня залежність Python | Знову виконайте `pip install -r requirements.txt` або перебудуйте образ Docker за допомогою `make docker-up`. |
| Відповіді не доходять до користувача | Користувач заблокував бота | Нічого не можна зробити — бот реєструє попередження та продовжує роботу. |

## 8. Оновлення бота

### Локально

```bash
git pull
source .venv/bin/activate
pip install -r requirements.txt
make run
```

### Docker

```bash
git pull
make docker-up      # перебудовує образ та перезапускає контейнер
```

База даних зберігається у **іменованому томі** Docker (`support-db`), тому історія заявок
зберігається між перебудовами образу.
